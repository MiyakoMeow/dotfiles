## 通用规则

### 代码风格

- 减少代码嵌套层数：尽可能将负面条件前置处理，错误分支提前返回。

### 代码引用

- 引用具体函数或代码片段时，使用 `文件路径:行号` 格式便于用户定位源码。

## 工作流编排

### 1. 计划节点默认
- 对于任何非平凡任务（3+ 步骤或架构决策），进入计划模式。
- 如果出现问题，立即停止并重新规划——不要继续硬推。
- 使用计划模式进行验证步骤，而不仅仅是构建。
- 编写详细规格以减少歧义。

### 2. 子代理策略
- 大量使用子代理以保持主上下文窗口整洁。
- 将研究、探索和并行分析任务卸载给子代理。
- 对于复杂问题，通过子代理投入更多计算资源。
- 每个子代理专注一个任务以实现聚焦执行。

### 3. 自我改进循环
- 在用户进行任何纠正后：将模式更新到 `tasks/lessons.md`。
- 为自己编写规则以防止同样的错误。
- 无情地迭代这些经验教训，直到错误率下降。
- 在会话开始时回顾与当前项目相关的经验教训。

### 4. 完成前验证
- 未经证明有效，绝不要标记任务为完成。
- 在相关时对比主分支与你更改后的行为差异。
- 问自己："资深工程师会批准这个吗？"。
- 运行测试、检查日志、演示正确性。

### 5. 追求优雅（平衡）
- 对于非平凡变更：暂停并问自己"有没有更优雅的方式？"。
- 如果修复感觉像补丁："基于我现在所知的一切，实现优雅的解决方案"。
- 对于简单、明显的修复可跳过此步——不要过度工程化。
- 在提交成果前先挑战自己的工作。

### 6. 自主修复 Bug
- 当收到 bug 报告时：直接修复它。不要请求手把手指导。
- 指向日志、错误、失败的测试——然后解决它们。
- 用户零上下文切换需求。
- 无需被告知方法即可修复失败的 CI 测试。

## 任务管理

1. **先计划**：将计划写入 `tasks/todo.md`，包含可检查的条目。
2. **验证计划**：开始实现前进行确认。
3. **跟踪进度**：随进度标记条目为完成。
4. **解释变更**：每一步提供高层摘要。
5. **记录结果**：在 `tasks/todo.md` 中添加回顾部分。
6. **捕获经验**：纠正后更新 `tasks/lessons.md`。

## 核心原则

- **简洁优先**：使每次变更尽可能简单。影响最小化的代码。
- **拒绝偷懒**：查找根本原因。不使用临时修复。遵循资深开发者标准。
- **最小影响**：变更应仅触及必要部分。避免引入新 bug。
